hypothesis: A per-environment baseline of legitimate kernel driver loads and driver-service installs (driver path, hash, signer, service name) makes anomalous BYOVD driver loads — like the Gentlemen group's GentleKiller framework dropping vulnerable/malicious drivers as services to terminate security software — visible as deviations from the approved set.
notes: 'Windows. Collection: Sysmon EID 6 (DriverLoad) capturing ImageLoaded (driver path), Hashes (SHA256/IMPHASH), Signature (signer/subject), Signed, SignatureStatus; Windows Security EID 7045 and EID 4697 (service install) capturing ServiceName, ImagePath, ServiceType (flag kernel-mode driver = 0x1), StartType, and the installing account/process; Sysmon EID 13 (registry) for writes under HKLM\SYSTEM\CurrentControlSet\Services\<svc>\ImagePath|Type|Start; Sysmon EID 1 / Security 4688 for the installing process lineage (sc.exe, rundll32, custom EXE). Enrich every record with host, user, and host role. Collection window: typically 30 days (driver loads are low-volume, so 30d gives a stable per-role picture; consider 60-90d for rarely-rebooted servers). Deliverable: an allowlist of approved signed drivers/driver-services per host role (workstation, server, DC, VDI, dev box) — each entry = driver path + hash + expected signer + expected service name + expected installing process; anything not on the allowlist is a hunt lead, feeding paired Flames [[H194]]. Immediate flags (do not wait for baseline): unsigned drivers (Signed=false); drivers whose SignatureStatus is Invalid/Expired/Revoked (enable Sysmon <CheckRevocation/>); drivers in user-writable paths (%TEMP%, %PROGRAMDATA%, %PUBLIC%, %APPDATA%, user profile) rather than %SystemRoot%\System32\drivers; known-vulnerable/malicious drivers matching LOLDrivers (e.g. dmx.sys CVE-2022-42045, IMFForceDelete CVE-2025-26125, nseckrnl.sys, GameDriverX64.sys, 360netmon_wfp.sys, googleApiUtil64.sys, ThrottleBlood.sys, havoc.sys, eb.sys, G11.sys); kernel-mode services (ServiceType 0x1) installed by a console/user EXE or by sc.exe with binPath outside System32\drivers; driver whose stated vendor (version info) does not match its signer cert (cross-ref T1036.001). Cross-ref T1562.001 (the driver is loaded to disable security tools) and paired Flames [[H194]] (the service-install + load chain) and [[H195]] (invalid/revoked signature).'

## Why
- BYOVD EDR-killers like GentleKiller (8+ variants, swapping among `eb.sys`, `dmx.sys`, `GameDriverX64.sys`, `IMFForceDelete`, etc.) defeat name/hash blocklists by rotating drivers, so the durable detection is "this driver/driver-service is not in our approved set," which requires a baseline — hence Embers, not Flames.
- Kernel driver loads and driver-service installs are genuinely low-volume in healthy environments, so a 30-day per-role baseline is cheap to build and produces a tight allowlist with few false positives.
- Several anomalies need no baseline at all (unsigned/revoked drivers, user-writable paths, LOLDrivers hits, vendor/signer mismatch) and should fire immediately, giving fast wins while the baseline matures.
- The baseline directly enables paired Flames hunt [[H194]] (service-install -> driver-load chain) and complements [[H195]] (signature status), turning a noisy raw event stream into a high-signal deviation feed before EDR is blinded.
