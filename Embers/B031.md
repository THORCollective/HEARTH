title: Baseline legitimate SNMP managers and config-backup flows to surface rogue MIB/config dumps
hypothesis: >-
  FSB Center 16 and similar actors dump network-device information and configuration over SNMP (weak/default
  community strings, SNMPv1/v2c) — but SNMP polling and scheduled config backups are also constant, legitimate
  management activity, so a static "SNMP to a router" rule is pure noise. This hunt first baselines normal SNMP
  management per device (which manager IPs poll which devices, with which SNMP version, over which OIDs, and which
  hosts legitimately pull configs to where), then flags the anomalies a MIB/config dump produces: SNMP from a source
  outside the management allow-list, reads of sensitive/config-copy OIDs from unexpected sources, SNMPv1/v2c where
  v3 is expected, and config transfers to destinations no backup job uses.
notes: >-
  PLATFORM: Network Devices (SNMP-managed routers/switches). BASELINE FIRST (Embers). Source: CISA/NSA/FBI joint
  advisory AA26-194A, "Improve Router Hygiene to Protect Against Russian State-Sponsored Targeting" (2026-07-13).
  The advisory's core mitigations — MIB allow-list, management ACLs, SNMPv3 authPriv, out-of-band management — are
  exactly the "normal" this baseline measures, so building it doubles as hardening evidence.


  DATA COLLECTION FIELDS (per SNMP/management event): source (manager) IP, destination device, SNMP version
  (v1/v2c/v3), community string or v3 user (record presence of weak/default strings like `public`/`private`),
  request type (Get/GetNext/GetBulk/Set), OID(s) accessed — especially CISCO-CONFIG-COPY-MIB (1.3.6.1.4.1.9.9.96)
  and other config/sensitive MIBs, request volume/rate, source port, timestamp; and for config backups: source
  host, destination server, protocol (TFTP/SCP/FTP), filename, direction. TELEMETRY: SNMP agent/server logs on
  managed devices, device syslog (config-change / config-copy notifications, CISCO-CONFIG-MAN-MIB
  `ccmHistoryEventConfigSource`), NetFlow/IPFIX or firewall flow at the management boundary (UDP/161-162 SNMP,
  UDP/69 TFTP, TCP/4786 Smart Install).


  COLLECTION WINDOW: 30 days per device to learn the stable, small set of legitimate managers, the SNMP version and
  OID scope each uses, and the normal config-backup source/destination pairs. DELIVERABLE: a per-device allow-list
  of expected SNMP managers (IP + version + OID scope) and expected config-backup flows (source NMS → destination
  server, protocol), plus an inventory of devices still permitting SNMPv1/v2c, default community strings, open
  UDP/69, or TCP/4786 — a direct remediation worklist.


  IMMEDIATE FLAGS (do not wait for the baseline to finish): (1) SNMP Set-Requests to any device, especially writes
  under CISCO-CONFIG-COPY-MIB `ccCopyTable` — legitimate management rarely uses SNMP Set to trigger config copies;
  (2) SNMP from a source IP not in the management allow-list, or from an external/spoofed-looking address; (3) reads
  of config/sensitive OIDs at high volume or from an unexpected source (MIB dump); (4) SNMPv1/v2c or default
  community strings on a device where v3 authPriv is expected; (5) an outbound TFTP/SCP/FTP config transfer from a
  device to a destination that appears in no known backup job. CROSS-REF: enables H235 (T1602.002 config
  exfiltration via SNMP-triggered TFTP copy) — that Flames hunt depends on this baseline to call an SNMP source
  "unexpected" or a transfer destination "first-seen." T1602.001 (this SNMP MIB dump) is the recon twin of the
  T1602.002 config dump.

## Why

- **SNMP is the busiest, most-trusted management protocol on the network fabric, so a signature fires on everything.** Only a per-device baseline of *which* managers legitimately poll *which* devices, with *what* version and OID scope, makes a rogue MIB dump or SNMP-triggered config copy stand out — which is why this is an Ember, not a Flame.
- **The advisory's own attack surface is a baseline gap.** FSB Center 16 succeeds precisely where SNMPv1/v2c and default community strings persist; inventorying who legitimately uses SNMP (and how) simultaneously produces the remediation list — devices to move to v3 authPriv and lock behind a management ACL.
- **The anomalies are structural, not IOC-based.** An SNMP source outside the management allow-list, a Set-Request under CISCO-CONFIG-COPY-MIB, or a config transfer to an unknown destination are deviations from a learned norm that survive the actor rotating infrastructure — durable signal no community-string block-list provides.
- **It is the prerequisite for the config-exfil Flame.** H228 can only call a TFTP destination "first-seen" or an SNMP source "unexpected" against this baseline of legitimate managers and backup flows; without it, the config-dump hunt has no normal to diff against.
