title: Baseline internal SMB/NTLM authentication to surface coerced/reflected auth (CVE-2025-33073)
hypothesis: >-
  Ransomware and hands-on-keyboard operators (e.g. The Gentlemen) abuse SMB-client coercion and NTLM reflection —
  including CVE-2025-33073 — to move laterally and escalate to SYSTEM. Because internal SMB/NTLM authentication is
  constant and mostly benign, a static rule is noise. This hunt first baselines normal internal SMB/NTLM auth per
  host and per account (who authenticates to whom, with what auth package, over signed or unsigned sessions), then
  flags the anomalies coercion produces: a host authenticating back to itself (loopback/reflected NTLM), machine
  accounts authenticating to unexpected peers, and SYSTEM-context SMB sessions to hosts a machine never normally
  reaches.
notes: >-
  PLATFORM: Windows / Active Directory. BASELINE FIRST (Embers). Source: Unit 42, "The Gentlemen Ransomware," which
  exploits CVE-2025-33073 (Windows SMB Client NTLM reflection EoP) and internal service vulns for lateral movement
  and recommends enforcing SMB signing + disabling SMBv1. This is a high-level brief with no forensic command
  lines, so the hunt is built as an authentication baseline rather than an IOC match.


  DATA COLLECTION FIELDS (per event): source host, destination host, account name, account type (user vs machine
  account `HOST$`), logon type (3 = network), authentication package (NTLM vs Kerberos), NTLM version, SMB dialect,
  SMB signing negotiated (yes/no), share/pipe accessed, process/service context, timestamp. TELEMETRY: Windows
  Security 4624 / 4625 (network logon, capture `Authentication Package` = NTLM and the source/destination), 4776
  (NTLM credential validation), 5140 / 5145 (share and detailed file-share access, esp. IPC$ and named pipes used
  by relay tooling — svcctl, atsvc, winreg), and SMBServer/SMBClient operational logs for dialect + signing.
  Sysmon EID 3 (network connect on 445) and EID 18 (named-pipe connect) enrich the picture.


  COLLECTION WINDOW: 30 days per host/account to learn normal peers, auth package, and signing posture. DELIVERABLE:
  a per-host/per-account allowlist of expected SMB/NTLM authentication — which accounts authenticate to which
  destinations, whether they use NTLM or Kerberos, and whether signing is enforced — plus a baseline of the small,
  predictable set of hosts that legitimately see machine-account inbound SMB.


  IMMEDIATE FLAGS (do not wait for the baseline to finish): (1) a host authenticating to ITSELF via network NTLM
  (source host == destination host, logon type 3) — the reflection tell; (2) a machine account (`HOST$`)
  authenticating to a host other than a domain controller or its normal management peers; (3) SYSTEM-context SMB
  sessions initiated to peers the machine never contacts, immediately followed by service creation (7045) or
  named-pipe access to `svcctl`/`atsvc`; (4) NTLM where Kerberos is expected (unsigned NTLM to an internal host
  that normally uses Kerberos), a hallmark of coercion/relay; (5) inbound SMB with signing NOT negotiated on hosts
  where policy requires it. CROSS-REF: enables lateral-movement Flames/Alchemy hunts (Impacket service-exec, PsExec
  `-s`, DCOM/WMI alternate lateral movement) and complements credential-access hunts on the DC.

## Why

- **Coercion and NTLM reflection hide inside the busiest protocol in the domain.** Internal SMB/NTLM authentication happens constantly and legitimately, so a signature on "NTLM to 445" fires on everything. Only a per-host/per-account baseline of *normal* auth makes the coerced or reflected session stand out — which is why this is an Ember, not a Flame.
- **CVE-2025-33073 turns a client into a SYSTEM foothold.** The SMB-client reflection vulnerability lets an attacker coerce a host into authenticating in a way that reflects back to SYSTEM, enabling local privilege escalation and lateral movement. The Gentlemen operators fold this into their intrusion chain, and Microsoft's fix pairs with enforcing SMB signing — the exact posture this baseline measures.
- **The anomalies are structural, not IOC-based.** A host authenticating to itself, a machine account reaching an unexpected peer, or NTLM where Kerberos belongs are deviations from a learned baseline that survive tool and infrastructure changes — durable signal a hash list can't provide.
- **The telemetry already exists and doubles as hardening evidence.** 4624/4776/5145 and SMB signing negotiation are already in most SIEMs; building the baseline simultaneously inventories where SMB signing is missing, directly reducing the relay/coercion attack surface.
