## Why

- The attacker's entire pivot from web-tier compromise to domain compromise hinged on a small set of file reads â€” server.xml and confluence.cfg.xml contained the `Jiraservices` domain-account credentials in plaintext, and reading those files cost the attacker nothing once they had SSH on the Confluence box (Microsoft, May 22, 2026)
- The detection signal is exceptionally clean once a baseline exists: the only legitimate readers are the JVM itself at startup, backup agents, and configuration-management runs â€” every other read is a candidate for credential theft. Without the baseline, every `cat server.xml` from an admin troubleshooting session is a false positive and the rule gets tuned to death
- The same baseline doubles as a detection layer for the broader pattern: any vulnerability that yields arbitrary file-read on a Confluence/Jira/Bitbucket host (CVE-2022-26134, CVE-2023-22515, CVE-2023-22518, CVE-2024-21683, future CVEs) ends in the same place â€” reading these specific files â€” so the baseline pays back across many incidents rather than just the F5/Confluence one
- Pairing the baseline with `[[H166]]` (F5â†’Linux SSH pivot) closes the chain end-to-end: the SSH origin signal tells you the session is anomalous, the config-file read signal tells you what they're after, and the post-read network activity tells you whether the credentials made it out of the host
