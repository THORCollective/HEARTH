# H234

Microsoft's July 2026 report on **ShinyHunters (UNC6040)** OAuth abuse documents a data-theft chain that never touches a vulnerability: operators vish employees while posing as IT support and guide them through Salesforce's OAuth consent screen to authorize an attacker-controlled connected app masquerading as **Salesforce Data Loader**. Once consented, the app makes API calls *as the user* — enumerating and exfiltrating CRM data through a trusted, approved integration. Because the activity rides a real identity and an approved app, sign-in and authentication monitoring "barely registers it." HEARTH had no coverage for **T1671 Cloud Application Integration**, and the durable hunting angle is the connected-app dimension: a first-seen app making API calls the user has never made before.

## Why

- **The abuse hides behind legitimate trust.** No malware, no failed logins, no impossible travel — just an approved OAuth app calling the API. The only place the anomaly surfaces is the connected-app attribution (`CONNECTED_APP_ID` + `UncommonForUser`), which is exactly what this hunt keys on.
- **The consent step is user-driven and self-identifying.** A non-admin end user consenting to a "Data Loader"-named app right after an inbound support call is a high-fidelity behavioral sequence that survives infrastructure rotation — operators change IPs and app names, but the consent-then-enumerate shape persists.
- **It is a persistence mechanism, not a one-shot.** The OAuth token grants standing API access that outlives the phishing call and any password reset, which is why T1671 sits under Persistence — killing the session does nothing; the connected app must be revoked.
- **The telemetry is now first-class.** Defender's upgraded Salesforce connector ingests Real-Time Event Monitoring with connected-app attribution and granted OAuth scopes, so the `ConnectedAppId`/`UncommonForUser` join the report ships is directly queryable rather than theoretical.
