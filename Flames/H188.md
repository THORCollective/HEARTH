# H188

An adversary with compromised AWS credentials or roles disables, deletes, or reconfigures CloudTrail to blind defenders immediately before privilege escalation, S3/KMS access, or data exfiltration. Because these are discrete, named management-plane API actions, the abuse is huntable directly in CloudTrail's own audit record ‚Äî specifically `StopLogging`, `DeleteTrail`, `UpdateTrail`, and `PutEventSelectors` events originating from non-console actors. The hunt looks for these log-tampering operations and pivots to the actor's subsequent high-risk activity within the same session.

## Why
- Unit 42's "Blinding the Watchmen" documents exactly these AWS primitives ‚Äî `stop-logging`, `delete-trail`, `update-trail` (KMS/destination changes), `create-trail` for log routing, and S3 `delete-bucket` for destroying log storage ‚Äî as a defense-evasion playbook, making them high-value hunt targets.
- Every one of these is a discrete, named CloudTrail `eventName`, so the abuse is recorded in the very service being attacked (until it isn't) ‚Äî ideal for a precise Flames hunt rather than a behavioral baseline.
- Filtering to non-console `userAgent`/`invokedBy` cuts the dominant false-positive source (admins toggling trails from the console) and surfaces programmatic, credential-driven tampering, which is the attacker pattern.
- The pivot from the tampering event to the same actor's subsequent IAM/S3/KMS activity converts a single suspicious API call into an attack-chain narrative, raising fidelity and giving responders immediate scoping.
