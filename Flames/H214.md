# H214

During the June 2026 DFIR Report Akira intrusion, operators who arrived via Bing/SEO-poisoned search (opmanager[.]pro → trojanized `ManageEngine-OpManager.msi` → `consent.exe` DLL-sideloading the BumbleBee loader `msimg32.dll` → AdaptixC2) executed commands using deliberate syntactic obfuscation: mixed-case interpreter invocation (`CmD.eXe`, `pOWerShELl.exE`) to defeat case-sensitive signatures, and Base64-encoded PowerShell run via WMI (`powershell.exe -e JABQAG8...`) to reconstruct and hide a DPAPI/Veeam-credential-theft script at runtime. Hypothesis: an adversary in this environment is masking malicious command content — through `-e`/`-EncodedCommand` Base64, case-toggling of `powershell`/`cmd`, and runtime string reconstruction (`Invoke-Expression`, `-join`, `[char]`) — to evade command-line signature detection, observable in PowerShell Script Block Logging and full-command-line process telemetry even though the final decoded script is what the engine logs.

## Why
- Directly models the observed Akira tradecraft: Base64 `-e` PowerShell executed via WMI and mixed-case `CmD.eXe`/`pOWerShELl.exE` are concrete command-obfuscation behaviors from the 2026-06-29 DFIR Report, not generic theory.
- EID 4104 Script Block Logging defeats the obfuscation by design — it records the deobfuscated script the engine actually ran — so this is a high-fidelity, low-cost hunt where logging is enabled, with 4688/Sysmon 1 catching the outer encoded launcher.
- Obfuscation is an interpreter-agnostic evasion layer sitting on top of T1059.001/T1059.003; hunting the encoding/case/reconstruction indicators catches BumbleBee/AdaptixC2 loaders and any commodity actor reusing `-enc`, `Invoke-Obfuscation`, or `Invoke-DOSfuscation`.
- Entropy/character-ratio scoring separates malicious obfuscation from benign long command lines, keeping the hunt tunable against known-good encoded-command tooling (SCCM, Defender, GPO scripts).
