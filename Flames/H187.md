# H187

OceanLotus (APT32) repeatedly abuses *legitimately signed* Windows executables — renaming `Toolbox.exe` to Genuine.exe/Updater.exe/AutoCAD242.exe and copying `dtlupdate.exe` to IntelAudioService.exe — and invokes them with the `-uiDll` command-line parameter to side-load a malicious, unsigned DLL (the SPECTRALVIPER downloader) from a non-standard path. The trusted, valid Authenticode signature on the parent EXE bypasses signature-based trust controls while attacker code executes inside the signed process. Hunt for valid-signed binaries that have been renamed away from their original filename and that load an unsigned/invalid-signature DLL from outside `%SystemRoot%`/`Program Files`, especially when the `-uiDll` parameter is present.

## Why
- ESET's report shows OceanLotus uses the trust of valid Authenticode signatures on `Toolbox.exe` and `dtlupdate.exe` to launch SPECTRALVIPER — the signature is real, so allow-by-signature controls pass while malicious code runs.
- The `-uiDll` parameter is the side-loading trigger and is an unusually specific, high-signal artifact to pivot on in command lines.
- Renaming a signed binary breaks the link between its PE `OriginalFileName` and its on-disk name — a cheap, reliable anomaly to compute from Sysmon EID 1.
- Pairing a valid-signed parent with an *unsigned* child DLL loaded from a user-writable path (Sysmon EID 7) is the core defeat of code-signing trust and rarely occurs benignly outside standard install paths.
