# H136

An adversary is reflectively loading code into memory on macOS using NSCreateObjectFileImageFromMemory or dlopen from memory-mapped regions to bypass Gatekeeper and file quarantine, indicated by processes loading unsigned Mach-O bundles without corresponding file creation events on disk.

## Why

- In the Sapphire Sleet macOS campaign, the icloudz backdoor was loaded via NSCreateObjectFileImageFromMemory — this macOS-specific reflective loading API bypasses Gatekeeper, file quarantine, and notarization entirely because the code never exists as a file on disk
- macOS reflective loading is fundamentally different from Windows reflective loading (H129) — it uses Mach-O dyld APIs rather than PE loading, requires different telemetry sources (ESF vs Sysmon/ETW), and bypasses different security controls (Gatekeeper vs AMSI)
- Endpoint Security Framework events for memory mapping (ES_EVENT_TYPE_NOTIFY_MMAP) provide the primary detection surface on macOS — traditional file monitoring will never see these payloads because they are loaded directly from network-received data in memory
- The combination of network data receipt followed by executable memory mapping without intervening file creation is a high-fidelity behavioral pattern that catches reflective loading regardless of the specific payload or encryption method
