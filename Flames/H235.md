# H235

Joint advisory **AA26-194A** (CISA/NSA/FBI plus 16 international partners, 2026-07-13) warns that **Russian FSB Center 16** is opportunistically compromising internet-facing routers by abusing SNMP. The actors scan for devices accepting weak/default community strings, then send SNMP Set-Requests — frequently from spoofed sources — writing **CISCO-CONFIG-COPY-MIB** OIDs that command the device to copy its own running configuration off-box via TFTP to an actor-controlled server. The stolen config yields credentials, topology, and ACLs for deeper intrusion. HEARTH had no coverage for **T1602.002 Network Device Configuration Dump**; this hunt targets the device-side and flow-side evidence of the SNMP-triggered config copy and its outbound exfiltration.

## Why

- **Config theft is a credential-and-map heist, not just recon.** A device's running-config carries hashed/typed passwords, SNMP community strings, VPN secrets, ACLs, and routing topology — everything needed to pivot deeper — so catching the copy-and-exfil early denies the actor their next several steps.
- **The behavior is detectable precisely because devices shouldn't do it.** Routers and switches almost never *initiate* outbound file transfers; an SNMP-sourced config copy followed by an outbound TFTP/FTP push from an infrastructure device is a sharp, low-base-rate signal once legitimate NMS backups are baselined out.
- **The OIDs are self-identifying.** CISCO-CONFIG-COPY-MIB writes (`ccCopyProtocol` = tftp, `ccCopyServerAddress` set to an external host) are a specific, high-fidelity indicator of remotely triggered config export that survives changes in the actor's infrastructure.
- **It aligns with a hardening posture defenders can verify.** The same advisory prescribes SNMPv3 authPriv, management ACLs, and disabling Smart Install — so building this hunt simultaneously surfaces the misconfigured devices (SNMPv1/v2c, open UDP/69, TCP/4786) that make the attack possible.
