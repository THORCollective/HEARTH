# H197

FishMonger's upgraded SprySOCKS for Windows persists by registering a malicious print processor that the Print Spooler service (`spoolsv.exe`) loads automatically at boot. ESET observed the actor creating the print processor `VSPMsg` with a `Driver = VSPMsg.dll` value under `HKLM\SYSTEM\ControlSet001\Control\Print\Environments\Windows x64\Print Processors\VSPMsg`, placing the DLL at `C:\Windows\System32\spool\prtprocs\x64\VSPMsg.dll`, and staging an encrypted payload at `C:\Windows\System32\spool\drivers\color\config.dat`. The loader explicitly checks that it was launched by `spoolsv.exe` and exits otherwise — both a sandbox-evasion gate and a tell that execution runs in the SYSTEM-level spooler context. We hypothesize that hunting for new sub-keys/`Driver` values under `...\Control\Print\Environments\*\Print Processors\*` paired with `spoolsv.exe` loading a non-standard DLL out of `\spool\prtprocs\` (or any unsigned/unknown module) will reliably surface this persistence and the broader print-processor abuse class, even as the processor name and DLL change.

## Why
- ESET ties this to FishMonger (I-SOON) SprySOCKS for Windows — a fresh, state-aligned China-nexus capability — making print-processor persistence current, attributed tradecraft.
- Print processors execute inside `spoolsv.exe` at SYSTEM, so this is both stealthy persistence and an effective privilege/context anchor; benign additions to the Print Processors key are rare, giving a clean registry-write signal.
- The two-event correlation (registry `Driver` write + `spoolsv.exe` loading a non-baseline DLL from `\spool\prtprocs\`) is durable: the processor name `VSPMsg` and DLL can rotate, but the load surface and directory do not.
- The loader's "only run under `spoolsv.exe`" gate is a deliberate sandbox-evasion choice that also defines the lineage defenders should hunt, making SYSTEM-context spooler children a high-fidelity pivot.
