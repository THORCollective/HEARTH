# H198

After establishing itself inside the SYSTEM-level Print Spooler context, FishMonger's SprySOCKS Windows loader manipulates access tokens to launch a backdoor under a stolen security context. ESET observed the loader (running as a print processor inside `spoolsv.exe`) obtaining a token from `spoolsv.exe` and calling `CreateProcessAsUserW` to spawn a new `svchost.exe`, into which it injects the backdoor shellcode â€” masquerading the malicious worker as a routine service host while running with a duplicated SYSTEM token. We hypothesize that hunting for token-theft process creation â€” a non-standard parent (e.g., `spoolsv.exe`) spawning `svchost.exe`/other LOLBins, correlated with `4624` Logon Type 9 (NewCredentials), `4672` special-privilege assignment (`SeDebugPrivilege`/`SeImpersonatePrivilege`), and process-creation events where the new process runs under a different LogonId/SID/integrity than its parent â€” will surface this `DuplicateTokenEx`/`CreateProcessWithTokenW`/`CreateProcessAsUserW` token-abuse chain regardless of which target process is spawned.

## Why
- ESET documents this as live FishMonger (I-SOON) SprySOCKS Windows behavior: a token taken from `spoolsv.exe` and used with `CreateProcessAsUserW` to launch and inject into `svchost.exe` â€” concrete, attributed tradecraft to hunt now.
- The technique combines token theft with masquerading and injection, so the cross-context signal (child running under a different LogonId/SID/integrity than its true parent) is more durable than any single binary or hash.
- `svchost.exe` not parented by `services.exe` (or lacking a valid `-k` service group) is an uncommon, high-fidelity anomaly that pairs cleanly with the token-theft chain to cut false positives.
- Correlating 4624 Type 9 + 4672 (`SeImpersonatePrivilege`/`SeDebugPrivilege`) + process creation gives a reliable detection signature for `make_token`/`DuplicateTokenEx` abuse that generalizes across actors beyond this campaign.
