## Why

- In the Sapphire Sleet campaign, the attacker installed a Launch Daemon named com.google.webkit.service.plist in /Library/LaunchDaemons — this provided root-level persistence that survives reboots and user logouts, and the naming convention was designed to blend in with legitimate Google services
- Launch Daemons run as root and execute before user login, making them the highest-privilege persistence mechanism on macOS — detecting unauthorized daemon installation is critical for identifying compromised hosts
- Legitimate Launch Daemon installations are rare in normal operations and typically only occur during software installation by signed packages — monitoring /Library/LaunchDaemons/ for new file creation provides a high-fidelity alerting surface
- The combination of masquerading names (mimicking Apple/Google) with unsigned or ad-hoc signed binaries creates a reliable detection pattern that catches this campaign and similar macOS implants
