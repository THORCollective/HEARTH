# H107

An adversary is clearing Windows event logs using wevtutil, Clear-EventLog, or similar utilities to destroy forensic evidence of their intrusion activity after achieving their objectives.

| Hunt # | Hypothesis | Tactic | Notes | Tags | Submitter |
|--------|-----------|--------|-------|------|-----------|
| H107 | An adversary is clearing Windows event logs using wevtutil, Clear-EventLog, or similar utilities to destroy forensic evidence of their intrusion activity after achieving their objectives. | Defense Evasion | Hunt for Event ID 1102 (Security log cleared), Event ID 104 (System log cleared), wevtutil.exe cl executions, and PowerShell Clear-EventLog cmdlet usage. Also look for gaps in event log continuity or event logs with abnormally small file sizes. | #defense_evasion #T1070_001 #event_log_clearing #anti_forensics | [Lauren Proehl](https://x.com/jotunvillur) |

## Why

- Event log clearing is one of the final actions ransomware operators take before or after deployment — in the ActiveMQ-to-LockBit intrusion, the adversary cleared System, Application, and Security event logs to hinder incident response
- Windows generates a meta-event (Event ID 1102) when the Security log is cleared, providing a reliable detection signal even after the logs themselves are destroyed — this is a high-fidelity indicator that is rarely triggered by legitimate administrative activity
- Adversaries who clear logs often do so selectively (e.g., clearing Security but not PowerShell Operational), so correlation across log channels can reveal the clearing activity and help reconstruct the attack timeline
- Log forwarding to a SIEM means cleared local logs can still be analyzed, but detection of the clearing action itself signals active adversary awareness of the environment's logging capabilities
