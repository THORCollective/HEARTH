# H189

In Google Cloud, an adversary with sufficient IAM permissions deletes, disables, or redirects Cloud Logging sinks and deletes log buckets to stop logs from being routed and stored — the GCP equivalent of the AWS CloudTrail tampering in [[H188]]. These are discrete operations recorded in GCP Cloud Audit Logs (Admin Activity) with named `protoPayload.methodName` values, so the log-router and log-storage abuse is huntable directly. The hunt targets sink deletion/disabling/redirection, log-bucket deletion, and CMEK impairment, then pivots to the responsible principal's surrounding activity.

## Why
- Unit 42's report enumerates the GCP-side blinding primitives directly — deleting log routers (`DeleteSink`), disabling a sink (`disabled=true`), redirecting routing (`logging.sinks.update` destination change), deleting log storage (`logging.buckets.delete`), and CMEK impairment — establishing these as concrete hunt targets, not theory.
- Each maps to a named `protoPayload.methodName` in GCP Cloud Audit Logs, so this is a precise Flames hunt against the log router/storage control plane rather than a behavioral baseline.
- `UpdateSink` with `disabled=true` or a changed `destination` is a quieter evasion than outright deletion — explicitly inspecting the request body catches the stealthier variant that a deletion-only rule would miss.
- Sister hunt [[H188]] covers the identical adversary behavior on AWS; running both gives multi-cloud coverage of T1562.008 with provider-native field paths and a consistent pivot-to-actor methodology.
