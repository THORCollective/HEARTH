# H208

Adversaries running device-code phishing initiate an OAuth 2.0 device authorization grant against Entra ID, then socially engineer a victim into entering the attacker-generated code at `microsoft.com/devicelogin` and approving the prompt. Because the victim authenticates legitimately (including MFA) on Microsoft's own portal, the attacker's polling backend silently receives valid access and refresh tokens â€” fully MFA-bypassing token theft with no credential compromise. In most enterprise tenants, the device-code flow is rarely used legitimately (it exists for input-constrained CLI/IoT/smart-TV scenarios), so its appearance is a discrete, high-signal event. This hunt searches Entra ID Sign-in logs for successful authentications where `authenticationProtocol == "deviceCode"` (and `originalTransferMethod == "deviceCodeFlow"` on subsequent token refreshes), then triages by user, location, client app ID, and resource to surface device-code grants that no legitimate workflow in the tenant should produce.

## Why
- Device-code phishing steals fully-formed OAuth tokens on Microsoft's legitimate portal, so it bypasses MFA and leaves no failed-credential or impossible-travel artifacts â€” the only reliable signal is the `deviceCode` authentication protocol value itself in Entra Sign-in logs.
- The device-code grant is near-zero in normal enterprise use (it targets input-constrained CLI/IoT devices), making its presence a discrete, low-false-positive Flames signal rather than something requiring volumetric baselining.
- Active, scaling threat: Storm-2372 popularized the technique and phishing-as-a-service kits (e.g. EvilTokens) have driven a sharp 2025â€“2026 surge, so most tenants are now in-scope targets.
- The telemetry this hunt queries (successful `deviceCode` sign-ins = the token-acquisition event), not the email link delivery, is why it maps to T1528 token theft rather than the spearphishing-link delivery technique.
