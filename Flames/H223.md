# H223

Mandiant (Google Threat Intelligence) documented an AD FS attack that recovers the **active** token-signing private key directly from the host's machine DPAPI / CAPI key store — not the well-known WID config DB + DKM path. When `AutoCertificateRollover` is disabled and an administrator manually rotated the signing certificate, the DKM/config path yields a stale, expired "ghost" certificate, while the live signing key sits protected only to SYSTEM in `MachineKeys`. With that key an adversary forges a SAML assertion impersonating a Global Administrator, which Entra ID accepts — full M365 tenant compromise with MFA and Conditional Access bypassed. HEARTH had no coverage for SAML-token forgery (T1606.x) at all.

## Why

- **Golden SAML defeats MFA and Conditional Access by design.** Once the token-signing key is stolen, the adversary mints tokens for any user without ever authenticating — so identity-plane token issuance, not endpoint logon, is where the hunt must live.
- **The active-key-from-MachineKeys path evades DKM-focused detections.** Hunts that watch the DKM container object (4662) miss this technique entirely because the live key never touches the DKM/AD object; file-level SACL (4663) on the machine key store is the sensor that catches it.
- **Orphaned token issuance is a high-fidelity, near-zero-FP signal.** A legitimate AD FS token cannot exist without a preceding primary authentication; a valid, Entra-accepted federated sign-in with no correlating AD FS auth is a forged token by construction.
- **The precondition is detectable as hygiene.** `AutoCertificateRollover` disabled plus a manually rotated cert (Event 385 warnings) marks the exact environments where the "ghost cert" gap exists — surfacing risk before an adversary uses it.
