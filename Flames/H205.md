# H205

After compromising a cloud tenant and obtaining only storage-deletion rights, an adversary deletes a victim bucket that is still referenced as a destination by an active data pipeline (CloudTrail/S3 replication, Amazon Data Firehose, GCP Cloud Logging sink, Pub/Sub, Storage Transfer Service, or Azure Monitor diagnostic settings), then immediately re-registers the same globally-unique bucket name in an attacker-controlled account. Because these pipelines resolve buckets by name rather than by immutable account ownership, the victim's own infrastructure keeps writing logs and sensitive data straight into the adversary's bucket — a silent Transfer Data to Cloud Account. We hypothesize that we can hunt this by finding bucket-deletion events for names that are still live pipeline destinations and that are NOT re-created within our own account/organization shortly after.

## Why
- The attack needs only low-privilege deletion rights (`storage.buckets.delete`/`DeleteBucket`/`Microsoft.Storage/storageAccounts/delete`), not the granular pipeline-edit permissions defenders usually watch, so a compromised Storage Admin role is enough — a realistic and under-monitored path.
- Post-hijack, the sink/replication config still validates as "healthy," so configuration review alone never surfaces it; the only durable signal lives in the deletion audit event correlated against live destination references.
- The deletion event is discrete, high-fidelity, and present in standard cloud audit logs (CloudTrail management events, GCP Cloud Audit Logs, Azure Activity Log) that mature teams already collect — low cost to operationalize.
- Impact is severe and stealthy: the victim's own trusted pipelines continuously exfiltrate logs and sensitive data to an external account, and log-sink hijacking simultaneously blinds the defender, so early detection of the delete is the best (and possibly only) intervention point.
