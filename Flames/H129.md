# H129

An adversary is reflectively loading encrypted PE payloads directly into memory to avoid disk-based detection, indicated by processes calling NSCreateObjectFileImageFromMemory (macOS) or Assembly.Load/Reflection APIs (.NET) without corresponding file writes.

## Why

- In the Elastic PhantomPulse campaign, the PHANTOMPULL loader reflectively loaded an AES-256-CBC encrypted PE payload entirely in memory, leaving no file artifact on disk for traditional AV to scan — this technique is increasingly common across both commodity and APT malware
- In the Sapphire Sleet macOS campaign, the icloudz backdoor was loaded via NSCreateObjectFileImageFromMemory, a macOS-specific reflective loading API that bypasses Gatekeeper and file quarantine entirely
- Reflective loading defeats disk-based scanning, file integrity monitoring, and hash-based detection — hunting for the API calls and memory patterns is the primary detection surface
- PowerShell Script Block Logging (4104) reliably captures .NET reflection patterns even when the assembly itself is obfuscated, making it one of the most effective data sources for this technique on Windows
