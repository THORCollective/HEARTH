title: Anomalous Salesforce/CRM bulk-export volume from a connected app (ShinyHunters data theft)
hypothesis: >-
  After obtaining API access through a malicious or compromised OAuth connected app, an adversary
  (ShinyHunters/UNC6040) bulk-exports CRM records — running large `ReportExport` and high-volume SOQL enumeration
  against high-value objects (Accounts, Contacts, Cases). Individual exports look like normal analyst activity, so a
  static threshold either misses slow exfiltration or drowns in false positives. Modeling each user's and each
  connected app's normal export/query volume makes the theft visible: a user or app whose export volume, object
  breadth, or query rate jumps far above its own learned baseline.
notes: >-
  ALCHEMY ANALYTIC — per-user / per-connected-app volumetric anomaly detection on CRM data access. PLATFORM: SaaS
  (Salesforce), surfaced via Microsoft Defender for Cloud Apps CloudAppEvents and Salesforce Event Monitoring /
  Event Log Files (ELF). Source: Microsoft, "Defending SaaS-based applications against ShinyHunters OAuth abuse"
  (2026-07-13), which notes the activity "appeared indistinguishable from legitimate Salesforce usage" and that
  effective detection requires spotting *anomalous* export/enumeration volume, not any single event.


  WHY MODEL-ASSISTED (not a static rule): report exports and SOQL queries are routine for sales/ops users, and
  ShinyHunters deliberately paces exfiltration ("slow exfiltration") to stay under fixed thresholds. The separating
  signal is the deviation from a given user's/app's own history — a volumetric/behavioral problem, not a signature.


  DATA SOURCES / FIELDS: CloudAppEvents where Application == "Salesforce", ActionType in
  ("ReportExport","ApiTotalUsage","UniqueQuery","API Event"); RawEventData fields `USER_ID`, `CONNECTED_APP_ID`,
  `CONNECTED_APP_NAME`, `QUERY_IDENTIFIER` (SOQL text/object), plus IPAddress, UserAgent. Salesforce ELF event types
  `ReportExport`, `API`, `BulkApi`, `RestApi` give row counts and durations for finer volume features.


  FEATURES per user AND per connected app over a rolling baseline (~30 days): count and summed row-volume of
  `ReportExport`; distinct high-value objects queried (Accounts/Contacts/Cases/Opportunities); SOQL query rate;
  ratio of read/export volume to that identity's historical mean; export activity outside business hours; export
  from an IP/user-agent first-seen for that identity; share of total-org export volume concentrated in one
  app/user. MODEL: per-entity baseline + robust outlier scoring (z-score / MAD or isolation forest) on the volume
  and breadth features; alert when an entity crosses from its normal envelope into sustained high-volume export,
  especially when the ConnectedAppId is itself first-seen (join to H227).


  ANCHOR QUERY (report-export volume by user, from the source report):
    CloudAppEvents
    | where Application == "Salesforce"
    | where ActionType == "ReportExport"
    | extend SalesforceUserId = tostring(RawEventData.USER_ID)
    | summarize Events = count() by AccountObjectId, AccountId, AccountName, SalesforceUserId, IPAddress, UserAgent

  VALIDATION / TUNING: backtest against known heavy-reporting periods (quarter-end, data-migration projects) to set
  thresholds; maintain per-entity drift; allowlist sanctioned ETL/integration service accounts by ConnectedAppId
  (but keep them baselined — a compromised vendor app, e.g. Salesloft/Gainsight, is exactly the supply-chain path in
  the report). NAIVE FOIL: a fixed "N exports/day" threshold — provably bypassed by paced exfil and noisy for
  power users; the separating signal is deviation from the entity's own baseline plus first-seen app/IP breadth.
  CROSS-REF: H234 (T1671) is the consent/foothold half — the connected app that consents there is the one whose
  export volume spikes here; also cross-ref T1567 (exfil over web service) for the outbound leg.

## Why

- **The event is legitimate; the volume is not.** Exporting a report and querying Contacts are things thousands of sales users do daily, so no single event is malicious. Only the deviation from an identity's own learned export/query envelope separates theft from work — a textbook per-entity anomaly-modeling problem, which is why this is an Alchemy hunt and not a Flame.
- **Static thresholds lose both ways.** ShinyHunters paces exfiltration to slip under fixed limits, so a low threshold is bypassed and a high one is blind; a per-user/per-app baseline adapts to each identity's normal and catches the *relative* spike a global rule can't.
- **Two entities must be modeled, not one.** Baselining the human user misses the case where a compromised third-party integration (Salesloft, Gainsight, Klue) does the exporting; baselining the connected app catches it. Modeling both — and joining on first-seen ConnectedAppId — is what makes the analytic robust.
- **The telemetry carries true volume.** Salesforce ELF and Defender's Real-Time Event Monitoring expose row counts, object names, and app attribution, so the model can score actual data volume and object breadth rather than a coarse event count — the difference between detecting a 10-row test and a full-CRM dump.
