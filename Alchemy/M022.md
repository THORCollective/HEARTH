hypothesis: After gaining a foothold, adversaries abuse the Microsoft Graph / legacy Azure AD Graph API to mass-enumerate the directory (users, groups, service principals, directory roles, applications), generating an anomalous spike in the volume and diversity of read/list calls from a single identity. Because some first-party apps and service principals legitimately call Graph heavily, a static call-count threshold is noisy; by fitting a per-identity (user or service principal) baseline over a rolling 30-day window and flagging anomalous enumeration breadth (distinct directory endpoints hit per hour) and burst volume, AzureHound/ROADrecon/AADInternals-style directory recon can be surfaced while legitimate heavy-Graph callers are tuned out.
notes: 'ALCHEMY ANALYTIC — per-identity baseline, not a static rule. Data source: Azure AD Graph Activity Logs / MicrosoftGraphActivityLogs table (Sentinel) or Elastic ingestion (ECS-mapped azure.aadgraphactivitylogs.*). Key fields: RequestUri/url.path, RequestMethod/http.request.method, ResponseStatusCode/http.response.status_code, ServicePrincipalId, UserId/user.id, IPAddress/source.ip, AppId (azure.aadgraphactivitylogs.properties.app_id), UserAgent/user_agent.original, api_version. Modeling features per identity (user OR servicePrincipalId) per hour: (1) count of DISTINCT directory endpoints hit across /users,/groups,/servicePrincipals,/directoryRoles,/applications,/devices,/policies,/organization,/tenantDetails — enumeration breadth; (2) total GET/list request volume vs that identity''s 30-day baseline (mean+stddev or MAD; flag z-score outliers / sudden 500+ request bursts — ROADrecon emits ~2,000 AAD Graph calls in ~1 min); (3) read/list-to-write ratio (recon is almost all GET, near-zero PATCH/POST/DELETE); (4) new identity suddenly enumerating — new_terms style: (calling OAuth client app_id, signed-in user) pair unseen in prior 14 days; (5) high 4xx error ratio from one caller (permission-probing); (6) suspicious UserAgents (python-requests, aiohttp, curl, Go-http-client, axios, AzureHound, BloodHound, AADInternals, roadrecon) — corroborating, not sole signal. ES|QL/KQL starting point on MicrosoftGraphActivityLogs: summarize dcount(RequestUri-resource) , count(), countif(ResponseStatusCode>=400) by UserId/ServicePrincipalId, AppId, bin(TimeGenerated,1h); join to a per-identity baseline table and emit rows where breadth or volume exceeds the identity''s own historical envelope. Article-confirmed enumeration patterns: ROADrecon gather walks every interesting object type (~2,000 calls/min via aiohttp); AADInternals dozens of cmdlets wrapping AAD Graph; AzureHound/BloodHound by user-agent; device-code sign-in to legacy graph.windows.net audience followed by directory enumeration within 5 min. Tuning: legitimate AAD Graph traffic is dominated by a small recognizable set of first-party callers (Microsoft.OData.Client, Microsoft Azure Graph Client Library, Microsoft ADO.NET Data Services, empty UA from first-party AppIds) — suppress/allowlist these known heavy-Graph apps and seed per-SP baselines so sync/provisioning service principals do not generate false positives.'

## Why
- The legacy Azure AD Graph Activity Logs (MicrosoftGraphActivityLogs) closed a decade-long blind spot: per-request telemetry (RequestUri, method, status, identity, app_id, IP, user-agent) finally makes mass directory enumeration observable, but the signal is buried in heavy legitimate first-party Graph traffic — exactly the conditions an analytic, not a static rule, is built for.
- Recon tooling is volumetric and broad by nature: ROADrecon's `gather` walks every interesting object type (users, groups, service principals, roles, apps) and emits ~2,000 AAD Graph calls in ~1 minute. The discriminating features are enumeration *breadth* (distinct endpoints) and a *burst* relative to that identity's own history — both of which a static org-wide threshold either misses (busy SPs) or false-alarms on (the legit sync app).
- A per-identity 30-day baseline separates a CRM sync service principal that always reads /users in bulk from a user account or new OAuth client that has never touched /servicePrincipals and /directoryRoles before suddenly sweeping the whole directory — the read/list-to-write ratio and (app_id, user) new-pairing features sharpen this further.
- User-agent strings (AzureHound, aiohttp, AADInternals) are high-fidelity but trivially spoofable, so they are used as corroborating evidence and ROADrecon-fingerprint boosters rather than the primary detection — the behavioral anomaly model survives user-agent customization, which is an explicit ROADtools evasion feature.
